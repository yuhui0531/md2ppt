# PRD: 导入已有提示词并进入工作台精调

## 1. 背景
当前 `md2ppt` 的主链路是：

`上传原始 Markdown -> 生成大纲与逐页 prompt -> 工作台精调 -> 批量生图 -> 导出 PPT`

但实际使用中，部分用户已经在系统外准备好了每一页的生图提示词，不需要再经过“上传原始 Markdown 再生成 prompt”的流程。他们更需要的是一个快捷入口，把现成提示词导入系统后，继续在工作台里微调、做一致性处理、批量生图并导出 PPT。

## 2. 目标
提供一条新的导入链路，满足“已有逐页提示词”的使用场景：

`导入提示词 -> 自动结构补全 / 大纲提取 -> 工作台精调 -> 批量生图 -> 导出 PPT`

本需求的核心目标：

- 支持用户直接导入已有 prompt，而不是强制重新生成
- 导入后仍然进入现有工作台体系进行精调
- 导入后的结构补全 / 大纲提取过程要有可见进度条和阶段说明
- 保持后续批量生图和 PPT 下载链路不变

## 3. 非目标
本期不包含以下内容：

- 不支持 JSON 导入
- 不自动重写用户导入的 prompt 正文
- 不新建第二套独立编辑器或工作台
- 不支持导入后继续走“上传 Markdown 项目”的完整生成链路

## 4. 目标用户

### 用户类型
- 已经在外部工具中写好了逐页生图 prompt 的用户
- 从系统导出的 prompt ZIP 经过外部修改后，希望重新导入的用户
- 希望跳过“内容解析与 prompt 生成”阶段，只做精修和出图的用户

### 典型诉求
- 快速把一批 `.md` prompt 导入系统
- 在系统内继续人工修改 prompt
- 对整套 prompt 做统一风格检查
- 批量生图并导出 PPT

## 5. 用户故事

### 用户故事 1
作为一个已经准备好逐页 prompt 的用户，我希望直接上传一个 ZIP 或多个 `.md` 文件，这样我就不用重新从 Markdown 原文开始生成。

### 用户故事 2
作为一个导入 prompt 的用户，我希望导入后仍然进入工作台，这样我可以逐页修改 prompt、新增页、删页，并继续做一致性修正。

### 用户故事 3
作为一个导入 prompt 的用户，我希望系统在导入后自动补全页面结构信息，并且在这个过程中看到进度条和状态说明，这样我能知道系统正在处理什么，而不是误以为卡住。

### 用户故事 4
作为一个导入 prompt 的用户，我希望后续批量生图和导出 PPT 的操作与原有流程一致，这样我不需要学习一套新的使用方式。

## 6. 核心流程

### 6.1 入口
在项目管理页提供两个入口：

- 新建素材项目
- 导入提示词

在新建页中增加模式切换：

- 上传原始 Markdown
- 导入已有提示词

### 6.2 导入方式
本期支持两种导入方式：

- 一个 ZIP 文件
- 多个 `.md` 文件

限制规则：

- ZIP 与多个 `.md` 不能混传
- 只识别 `.md`
- 忽略 `index.md`
- 忽略隐藏文件、目录、`__MACOSX`、非 `.md` 文件

### 6.3 导入后行为
导入完成后系统应：

1. 创建一个“导入型项目”
2. 将每个 `.md` 文件映射为一页 slide
3. 自动触发“结构补全 / 大纲提取”后台任务
4. 立即跳转到工作台
5. 在工作台顶部展示该任务的进度条

### 6.4 工作台精调
导入项目在工作台中支持：

- 逐页浏览
- 修改 prompt
- 保存 prompt
- 新增页
- 删除页
- 一致性检查
- 修正不一致
- 进入批量生图页

### 6.5 生图与导出
导入项目后续流程与现有一致：

- 批量生图
- 查看图片
- 下载 PPT

## 7. 产品要求

### 7.1 导入项目定义
系统需要新增“导入型项目”概念，用于与现有“Markdown 生成型项目”区分。

建议使用字段：

- `project_origin = imported_prompts`

同时保留现有生成型项目，例如：

- `project_origin = generated_markdown`

### 7.2 导入项目初始数据
导入项目创建时，每一页至少具备：

- 页码
- 唯一 id
- 标题
- prompt 正文

页面结构相关字段允许先为空或填默认值，但必须在后续结构补全任务中补出。

系统需要在导入时直接初始化默认视觉规范，以保证一致性检查和后续生图链路可用。

### 7.3 结构补全 / 大纲提取
导入项目创建后，系统必须自动执行后台任务，用于从当前 prompt 中提取结构化信息。

任务目标包括：

- 逐页提取标题、页面类型、核心表达、版式建议等
- 汇总项目级大纲信息
- 保存结构化结果，供工作台展示与后续功能使用

关键约束：

- 该任务不能改写 `slide.prompt`
- 该任务的职责是“提取结构”，不是“重写 prompt”

### 7.4 进度条要求
“结构补全 / 大纲提取”任务必须展示进度条。

进度条要求：

- 有百分比进度
- 有阶段名称
- 有当前 message
- 刷新页面后能自动接续展示
- 失败时有明确错误信息

建议阶段包括：

- 任务已创建
- 扫描导入文件
- 解析逐页提示词
- 提取页面结构
- 汇总整体大纲
- 保存结构化结果
- 完成 / 失败

建议 message 示例：

- 正在解析第 3/12 页提示词
- 正在提取页面结构（8/12）
- 正在汇总整套提示词的大纲信息
- 正在保存结构化结果

### 7.5 工作台按钮策略
导入项目进入工作台后，不应完全复用“Markdown 生成型项目”的所有按钮语义。

需要保留：

- prompt 编辑
- 新增页 / 删除页
- 一致性检查
- 修正不一致
- 批量生图

需要隐藏：

- 继续生成
- 重生成当前页 prompt
- 重生成全部 prompt

原“重新生成大纲”按钮对导入项目应改为更准确的语义，例如：

- 重新解析页面结构
- 重新生成结构化大纲

该操作只更新结构化字段，不改写 prompt。

## 8. 文件解析规则

### 8.1 页面排序
导入的页面顺序按文件名排序：

1. 文件名前缀数字优先
2. 没有数字前缀时按文件名字典序

### 8.2 标题提取
每页初始标题取值优先级：

1. Markdown 第一条标题
2. 第一条非空文本
3. 文件名去扩展名

### 8.3 prompt 保存规则
每个文件全文直接保存到 `slide.prompt`。

系统不得：

- 自动清洗 prompt 内容
- 自动重写 prompt 内容
- 自动替换用户表达方式

## 9. 功能范围

### 本期必须完成
- 支持导入一个 ZIP
- 支持导入多个 `.md`
- 导入后自动创建项目
- 导入后自动触发结构补全 / 大纲提取任务
- 导入后直接进入工作台
- 工作台展示该任务进度条
- 工作台支持 prompt 精调
- 导入项目支持一致性检查
- 导入项目支持批量生图
- 导入项目支持导出 PPT

### 本期不做
- JSON 导入
- 自动重写 prompt
- 导入后自动开始生图
- 为导入项目提供“继续生成”能力

## 10. 验收标准

### 功能验收
- 用户可以上传一个 ZIP 成功创建项目
- 用户可以同时上传多个 `.md` 成功创建项目
- 导入后页面自动跳转到工作台
- 工作台顶部能显示结构补全 / 大纲提取进度条
- 进度条包含阶段名称和动态说明
- 导入后的原始 prompt 内容保持不变
- 用户可以在工作台修改 prompt 并保存
- 用户可以新增页、删除页
- 用户可以执行一致性检查与修正
- 用户可以进入批量生图并导出 PPT

### 体验验收
- 用户清楚知道系统正在处理什么
- 大批量导入时进度条持续推进，不长期停在 0%
- 刷新页面后任务进度不丢
- 任务失败时能看到明确错误
- 即使结构补全失败，用户仍可手动编辑 prompt

## 11. 风险与注意事项
- 用户导入的 prompt 格式差异会很大，结构提取准确性可能不稳定，因此必须保留人工精调能力。
- 导入项目没有原始 Markdown 语料，不适合复用“继续生成”一类基于原始内容的流程语义。
- 如果不对导入项目区分 `project_origin`，工作台容易暴露错误操作入口，导致用户理解混乱。
- 如果结构补全任务没有明确进度条和 message，用户会误判系统卡死。

## 12. 成功指标
- 导入成功率高，无格式误判导致的大面积失败
- 用户可在一次导入后顺利完成 prompt 精调并进入生图
- 导入项目的工作台使用路径与现有项目足够一致，不增加额外学习成本
- 导入后任务进度清晰，用户无需反复刷新猜测处理状态
