# Review：backend-generation-speedup-plan.md

整体方向对，核心诊断（串行、每页建连、每页落库）都和代码一致。但有几个地方需要补充或澄清：

---

## 问题一：并发改造缺少中间状态处理

计划说把 `generate_brief()` 和 `extract_source_slide_count_constraint()` 改为并发，但代码里两次调用之间有一次 `save_project_data()`（generation_service.py:112）。并发后这次中间保存怎么处理？是等两个都完成再一起保存，还是取消中间保存？需要明确，否则实现时会有歧义。

---

## 问题二：阶段 token cap 的注入机制未说明

计划说"在 `_call_json()` 增加阶段级 token cap"，但 `_call_json()` 本身不知道当前是哪个阶段。需要说清楚：是给 `_call_json()` 增加 `stage` 参数，还是在各调用点直接传 `max_tokens` 覆盖？不说清楚，实现者会做出不一致的选择。

---

## 问题三：节流落库的部分恢复语义被削弱了，但没有说明取舍

现在每张图完成后立刻持久化，是支持"进程崩溃后恢复"的设计。改成节流后，最多丢失 1 秒内完成的图片数据。计划说"保持部分成功语义"，但这个语义实际上是被弱化的（节流窗口内崩溃会丢数据）。应该显式说明这是可接受的取舍，以及节流窗口为什么是 1.0s 而不是更短或更长。

---

## 问题四：共享 GatewayClient 的 API 设计未说明

计划说"支持注入外部 AsyncClient"，但没说具体怎么注入：构造函数参数？工厂方法？context manager？生图链路和文本链路的注入点在哪里？`httpx.AsyncClient` 的连接池参数（`limits`, `timeout`）怎么配置？这些对实现影响很大，建议补充。

---

## 问题五：Token cap 值缺乏来源说明，也不可配置

`风格一致性检查` 的 cap 是 4096，`内容理解摘要` 是 3072，但计划里没说这些数字从哪来（实测？估算？）。另外计划为生图链路新增了两个 env var，但 token cap 值是硬编码，不一致。建议要么说明这些数字的来源，要么也做成可配置。

---

## 问题六：Prompt 改动和运行时改动风险不对等，但放在同一个测试计划里

运行时优化（连接复用、并发、节流）不影响输出内容。Prompt 改动直接影响输出质量，schema 合法不等于质量没有回归。当前测试计划只验证 schema 和结构完整性，没有输出质量的基准对比。建议补充：对同一份输入，优化前后的关键输出字段做人工抽样对比，或者至少保留一份 golden output 做回归。

---

## 小问题：优先级排序在 Assumptions 里，但没有对应的分阶段实施建议

计划末尾说"提示词优化优先级低于运行时改动"，但没有说这两部分是同一个 PR 还是分开。如果一次性全上，出了问题很难定位是哪部分导致的。建议明确是否分两个 PR，或者至少说明提示词改动可以独立回滚。

---

## 总结

| 类别 | 状态 |
|---|---|
| 问题诊断 | 准确，和代码一致 |
| 运行时优化方向 | 正确，但 GatewayClient 注入机制和 token cap 注入点需要补充 |
| 节流落库的语义取舍 | 需要显式声明 |
| Prompt 改动的质量验证 | 测试计划有盲区，需要补充 |
| 分阶段策略 | 建议补充 |

最核心的两条：**token cap 的注入机制**和**并发改造后的中间保存处理**，不说清楚实现时一定会卡住。
